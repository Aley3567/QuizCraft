# Agent Readiness

Every issue produced by `/to-issues` should include an `## Agent Triage` section:

```md
## Agent Triage

- Role: ready-for-agent | ready-for-human | blocked-by-dependency | needs-info | wontfix
- Type: type:prd | type:slice | type:bug | type:chore
- Reason: One concise sentence explaining the role.
- Dependencies:
  - None
  - #123
  - phase1.slice1.2
- Claimable now: yes | no
- Suggested first increment: The smallest useful first implementation increment.
- Verification:
  - `command or concrete check`
```

Every implementation issue must also include TDD fields:

- User behavior
- Public interface
- First red test
- Minimal green
- Acceptance criteria
- Out of scope

## Definitions

- `ready-for-agent`: the agent can work AFK now, dependencies are satisfied, and verification is local or mockable.
- `ready-for-human`: implementation or acceptance requires human judgment, credentials, external permissions, production ownership, or subjective review.
- `blocked-by-dependency`: the work is specified, but upstream code or issue dependencies are missing.
- `needs-info`: the work lacks enough detail or verification criteria.

`ready-for-agent` is not the same as "the issue was generated by `/to-issues`". Generated issues can be spec-ready while still blocked or human-owned.
